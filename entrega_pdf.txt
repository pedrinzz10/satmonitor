[DESCRIÇÃO DO PROJETO]

O SatMonitor é uma API REST para monitoramento de satélites em
missões espaciais. Operadores se autenticam via JWT (8 horas,
HMAC-256) e interagem com uma hierarquia de entidades:

  Agência → Missão → Satélite → Sensor → Leitura → Alerta

Quando uma leitura ultrapassa os limites configurados no sensor,
um alerta é gerado automaticamente. O controle de acesso é feito
por roles por missão (DONO, SUPERVISOR, MEMBRO).

[DESCRIÇÃO DA INFRAESTRUTURA]

A aplicação está hospedada em uma Azure Virtual Machine (Ubuntu
22.04, Standard D2s v3 — 2 vCPUs, 8 GB RAM) na região East US 2.

Recursos Azure provisionados:
  - Resource Group: rg-satmonitor
  - Virtual Network: vnet-satmonitor (10.0.0.0/16)
  - Subnet: subnet-satmonitor (10.0.1.0/24)
  - Network Security Group: portas 22 (SSH), 8080 (API), 5432 (DB)
  - Public IP: estático, Standard SKU
  - Network Interface Card: vincula IP público à VM
  - Virtual Machine: vm-satmonitor-RM562312
  - Managed Disk: OS 30 GB (implícito na criação da VM)

================================================================

[CONTAINERIZAÇÃO]

A aplicação usa Docker Compose com dois containers:

  satmonitor-app-RM562312
    - Imagem: build local via Dockerfile multi-stage
    - Base: eclipse-temurin:21-jdk (build) → eclipse-temurin:21-jre-alpine (runtime)
    - Porta: 8080
    - Usuário: satuser (sem privilégios root)
    - Healthcheck: GET /actuator/health

  satmonitor-db-RM562312
    - Imagem: postgres:16-alpine
    - Porta: 5432
    - Volume: satmonitor-db-data (persistência garantida)
    - Healthcheck: pg_isready

Rede interna: satmonitor-net (bridge)
Perfil: --profile docker (separa ambiente local H2 de produção)

================================================================

[CI/CD — GITHUB ACTIONS]

Pipeline em .github/workflows/ci-cd.yml:

  1. Trigger: push na branch main
  2. Job test: executa ./gradlew test no Ubuntu runner
  3. Job deploy (após test): conecta via SSH na VM Azure,
     executa git reset --hard + git pull + docker compose up --build -d
  4. Health check com retry (até 5 tentativas) após o deploy
  5. Notificação de falha caso algum step quebre

Secrets configurados no GitHub:
  - SSH_PRIVATE_KEY
  - VM_HOST
  - VM_USER

================================================================

[PERSISTÊNCIA DE DADOS]

O volume Docker satmonitor-db-data é gerenciado pelo Docker Engine
na VM e sobrevive a qualquer ciclo de stop/start dos containers.
Os dados permanecem mesmo após docker compose stop e restart.
